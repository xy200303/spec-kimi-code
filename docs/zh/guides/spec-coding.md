# 规格驱动开发

Spec Kimi 将开发记录视为项目的一部分。交互式会话从 Plan 模式开始，并在实现前创建运行目录，因此完成的变更可以被检查为一条由意图、审批、执行和证据组成的链路，而不是一份无法解释的 Diff。

## 工作流

工作流分为五个阶段，每个阶段都在进入下一阶段前收敛不确定性。

1. **定义工作**：明确目标、约束、范围和验收标准。
2. **规划与设计**：检查仓库，准备规格和设计，并识别风险及验证工作。
3. **审批快照**：在实现前审查计划结果。获批的规格和设计会建立快照，后续漂移会被检测。
4. **按任务执行**：完成带有必要性、类别、风险等级和可追踪变更的任务。受控操作会应用风险感知权限。
5. **验证与最终化**：附上证据，满足选定质量门禁，并最终化交付记录。最终化后的记录会被锁定。

## 项目内记录

每次运行都会在工作项目根目录使用一个目录：

```text
specs/<run-id>/
  spec.md
  design.md
  delivery.md
  delivery.json
```

`spec.md` 记录期望结果、约束和验收标准。`design.md` 记录任务拆解、影响范围、风险和验证方式。`delivery.md` 是面向人的交接记录，`delivery.json` 则供工具和自动化读取。

该位置是刻意设计的。文档可以在编辑器中审查，按项目策略提交，并与源代码变更对比；它们不会隐藏在用户级 Kimi Code 数据目录中。

## 策略路由

规格获批后，开发策略路由会根据已描述的工作选择策略。支持的策略为 `planning`、`agile_mvp`、`controlled_feature`、`bug_diagnosis`、`refactor`、`review`、`release` 和 `research`。

选定策略会记录选择原因、必须覆盖的任务类别和建议质量门禁。例如，故障诊断需要复现、根因和回归测试任务；发布建议使用 release 门禁，并要求发布构建和发布说明任务。未匹配专用信号时，使用受控功能开发流程。

## 任务与风险控制

每个任务记录其必要性、状态、类别和风险。执行轨迹会把工具调用、修改路径、命令和委托工作关联到当前任务，因此交付记录能够回答某个文件为何修改，以及它服务于已批准目标的哪一部分。

风险控制区分低、中、高风险任务。风险越高，审批路径越严格；审批决策会保留在运行记录中。目标不是追求最大自动化，而是让自动化始终可审查。

## 质量门禁与证据

Spec Kimi 支持四种质量门禁：`fast`、`standard`、`strict` 和 `release`。门禁决定运行被标记完成前必须具备的证据。证据可以包括聚焦测试、构建、类型检查、lint、Diff 审查，以及其他与所需任务关联的验证。

交付记录包含目标、约束、计划、任务、变更、证据、决策、风险、开放问题和回滚说明。缺少必需证据或策略要求的任务类别时，运行不能被最终化为完成。

## 使用 CLI

在项目中启动会话：

```sh
spec-kimi
```

交互式会话已经默认从 Plan 模式开始，因此 `--plan` 只是显式等价写法。需要审批和交付证据的实现工作不要使用 `-p`：它是非交互输出接口，无法呈现审查步骤。
